1Executive Overview

What the project is, the problem it solves, and the promises it must keep.

What is SignVault?

SignVault is a production-deployed web application that acts as a digital notary. An authenticated user uploads any file (PDF, image, document); the server computes the file’s cryptographic fingerprint and signs it with a private RSA key that only the server holds. The result is a tamper-evident signature bound to that exact file and to the person who requested it. Afterwards, anyone — with no account — can verify the file by dropping it into the public Verify page or by opening a share link, and instantly see who signed it and whether it has changed by even a single byte.

The problem it solves

Documents that travel between parties (contracts, agreements, letters, invoices) have no built-in proof of authenticity. A PDF can be edited and re-sent; a name at the bottom proves nothing. SignVault provides a verifiable chain of trust: the platform is a known authority, its public key is published, and a valid signature is mathematical proof that the document passed through it unchanged.

The three promises (these drive every design decision)

PromiseMeaningMechanism
IntegrityNot one byte has changed since signing.SHA-256 hash of the file bytes.
AuthenticityIt was signed by SignVault, not an impostor.RSA-3072 signature with a private key the server alone controls.
Non-repudiationThe action cannot later be denied.Append-only audit log with actor, action, IP and timestamp.

Core objectives

  • Be a real, usable product — not a toy: registration, signing, verification, sharing, certificates, and deletion, all working end-to-end and deployed publicly.
  • Be approachable yet professional — a clean, bilingual (English / Arabic, LTR & RTL) interface, and verification that requires no account so trust is frictionless.
  • Be portable and deployable — run on a laptop with zero database setup, yet ship to the cloud, to a container, or to Windows IIS with the same code.
  • Be portfolio-grade — the project was built as a demonstration of full-stack and DevOps competency aligned with an enterprise/government .NET + Angular role.
Current status The system is live at signvault-q8mi.onrender.com (Render, free tier), built and published as a public container image, with CI/CD and push-to-deploy wired, and a free custom domain (signvault.is-a.dev) in flight. It also runs locally and on IIS.

2What Was Implemented

The complete inventory of what was built, how it is structured, and how data flows.

High-level architecture

SignVault is a classic three-tier web application. In development the two halves run separately (Angular dev server proxies to the API). In production they are fused into a single origin: the ASP.NET Core API serves the compiled Angular app from wwwroot and exposes the API under /api, with a SPA fallback so client-side routes survive a refresh. This removes the need for CORS and a separate frontend host in production.

The signing engine

The heart of the product is the ISigner abstraction with an RsaSigner implementation backed by an X509Certificate2. The certificate is resolved at startup from one of three sources, in order of precedence: an environment variable (Signing__PfxBase64, for a stable cloud key), a PFX file on disk, or a freshly generated self-signed certificate. Signing uses RSA.SignData(bytes, SHA256, Pkcs1); verification uses the public key.

Services/ISigner.cs (essence)
public byte[] Sign(byte[] data)   => rsaPrivate.SignData(data, SHA256, Pkcs1);
public bool Verify(byte[] data, byte[] sig) => rsaPublic.VerifyData(data, sig, SHA256, Pkcs1);

Data flow — signing & verification

Signing (authenticated)

// inside one EF transaction:
load Document (owner check)
bytes = fileStore.Read(storageKey)
sig   = signer.Sign(bytes)
save Signature{ SignerName, base64, thumbprint }
doc.Status = Signed
audit.Record("DOC_SIGNED")
commit

Verify-by-file (public, no ID)

bytes = uploaded file
hash  = SHA256(bytes)
doc   = find signed doc WHERE ContentHash == hash
ok    = signer.Verify(bytes, doc.sig)
=> { valid, signerName, fileName, signedAt }

The verify-by-file lookup is what makes public verification effortless: a visitor simply drops the file and the system finds its signature by fingerprint — no document ID, no account.

Security & production hardening

  • Passwords: BCrypt hashes only (work factor 12).
  • JWT: validated issuer/audience/lifetime/signing key; ownership re-checked server-side on every document action.
  • Signing key: loaded from an environment variable in production (stable across redeploys); never committed.
  • Atomicity: a signature and its audit row commit inside a single EF Core transaction.
  • Edge hardening: forwarded-headers (real client IP behind the proxy), HSTS in production, security headers (X-Content-Type-Options, X-Frame-Options, Referrer-Policy), and a per-IP rate limiter (240 req/min).

DevOps & deployment (what was configured)

  • Dockerfile — multi-stage, single-origin image; writable /data for DB/key/uploads; honours $PORT; libfontconfig1 for PDF rendering.
  • GitHub Actions — ci.yml: build .NET, build Angular, vulnerability scan, publish image to ghcr.io/anas-lees/digital-signature-platform, and a deploy job that triggers Render via a secret hook. deploy-iis.yml: push-to-deploy to a self-hosted Windows runner.
  • render.yaml — one-click Blueprint that provisions a free PostgreSQL database, wires DATABASE_URL, and runs the Docker image.
  • deploy/ scripts — Publish.ps1, Deploy-IIS.ps1, the self-elevating Deploy-IIS-All.ps1, and New-SigningKey.ps1 (generates a stable signing key), plus DEPLOY.md.
  • Live deployments verified: Render (public URL), local single-origin (Kestrel), and Windows IIS (port 8090).

3Out of Scope & Omissions

Where the system deliberately stops, and the known technical debt an inheriting engineer must understand.

Intentionally out of scope

  • Legally-binding / qualified e-signatures. The platform uses a self-signed X.509 certificate, not one from a recognised Certificate Authority, and there is no qualified-signature (eIDAS / ESIGN) compliance. It is a demonstration of the cryptography, not a notarisation service.
  • PAdES (signature embedded inside the PDF). Signatures are stored alongside the document in the database, not embedded into the file. The PDF “certificate” is a separate human-readable receipt, not an in-file digital signature that Adobe Reader would show as a green banner.
  • Trusted timestamping (RFC 3161). Signed-at times are the server’s own clock; there is no independent timestamp authority proving when something was signed.
  • Identity assurance. Anyone can register with any email; there is no email verification, KYC, or federated/SSO identity. “Signed by Anas” means “signed by the account that called itself Anas.”
  • Multi-party / workflow signing. One owner signs their own document; there is no send-for-signature, counter-signing, or approval workflow.

Features & robustness not built

AreaWhat’s missing
Auth lifecycleNo refresh tokens or token revocation (access token only, 120-min lifetime); no password reset; no MFA; no account lockout on repeated failures.
AuthorizationAn Admin role exists in the model but no endpoint actually uses it — there is no admin dashboard.
UploadsOnly a 25 MB size limit. No file-type allow-listing, no content/MIME validation, no malware scanning.
ListingNo pagination, search, or sorting on the documents list — it returns everything for the user.
StorageFiles are written to local disk (or the container’s /data); no object storage (S3 / Azure Blob).
ObservabilityDefault ASP.NET Core logging only — no structured logging (Serilog), correlation IDs, metrics, or tracing wired into the running app.
TestingNo backend automated test project; 5 frontend unit tests only; no integration or end-to-end tests.

Technical debt & “honest” compromises

Schema management on Postgres SQLite uses proper EF Core migrations; Postgres uses EnsureCreated(), which builds the schema from the model on first run but cannot evolve it. Adding a column later means writing real Postgres migrations (or recreating the database). This was a deliberate speed-vs-rigor trade for the demo.
Audit log is convention, not enforced The audit table is append-only by code discipline, not by database permissions, and rows are not hash-chained. A compromised app could, in principle, alter history. True non-repudiation needs WORM storage or hash-chaining.
Ephemeral data on the free tier Unless PostgreSQL and a stable signing key (Signing__PfxBase64) are configured, a cloud restart resets the database and mints a new key, which invalidates previously created signatures. The wiring to fix this exists; it requires two environment variables to be set.
Minor stubs & conventions The Jwt:Key and PFX password have insecure development fallbacks (overridden by env vars in production); CORS is hard-coded to localhost dev origins (harmless because production is single-origin); the demo account credentials are public; and www.signvault.is-a.dev was not registered (only the apex subdomain).

4Engineering Trade-offs & Alternatives

For every major decision: what was chosen, what was considered instead, and why the alternative was rejected.

DataSQLite by default, PostgreSQL in production
Decision
Ship on SQLite (a single file, zero install) and auto-switch to PostgreSQL when a DATABASE_URL is present.
Alternatives
MySQL (the original brief) or SQL Server; running everything on a managed Postgres from day one.
Why
The build machine had neither MySQL nor Docker installed, so SQLite was the only friction-free path to “clone and run.” Because the app is written against EF Core and interfaces, the production swap is a one-line provider change — the entities, services, and entire frontend are untouched. This demonstrates the portability rather than just claiming it.
ArchitectureLayered monolith, not multi-project Clean Architecture
Decision
One SignVault.Api project organised into folders (Domain / Data / Dtos / Services / Controllers).
Alternatives
The textbook four-project Clean Architecture solution (Api / Application / Domain / Infrastructure).
Why
For a single developer and a runnable demo, four projects add ceremony without benefit. The folder structure still enforces separation of concerns and uses interfaces (ISigner, IFileStore, IAuditLogger), so it can be split into real projects later with minimal churn. The dependency direction is already correct.
Platform.NET 10 / Angular 20 — latest, not LTS-of-the-brief
Decision
Target the versions actually installed on the machine (.NET 10, Angular 20).
Alternatives
Pin to .NET 8 / Angular 17 to match common enterprise baselines.
Why
Using the installed SDKs avoided a download/setup detour and showcases current APIs (built-in OpenAPI, signals, new control-flow syntax). The concepts are identical across versions; the cost is that some third-party tooling lagged (see OpenAPI below).
API docsBuilt-in OpenAPI + Scalar, not Swashbuckle
Decision
Use .NET 10’s native AddOpenApi() plus the Scalar UI (development only).
Alternatives
Swashbuckle (Swagger UI), the long-standing default.
Why
On .NET 10, Swashbuckle collided with Microsoft.OpenApi 2.x (a moved namespace) and failed to build. Rather than fight version pinning, switching to the framework’s own OpenAPI generator plus Scalar was cleaner, modern, and conflict-free.
CryptoSign the file bytes; store the hash separately
Decision
SignData(bytes) / VerifyData(bytes); the SHA-256 hex is stored for display and for hash-based lookup.
Alternatives
Hash first, then SignHash / VerifyHash.
Why
Functionally equivalent, but SignData is one fewer moving part and harder to get wrong. The stored hash unlocks the killer feature — public “drop a file, find its signature” verification — without changing the security model.
Signing keyEnv-var PFX in production, generated key locally
Decision
Load the certificate from Signing__PfxBase64 when set; otherwise from a file; otherwise generate a self-signed one.
Alternatives
A hardware HSM / cloud Key Vault from the start; or always generating at runtime.
Why
An HSM is the correct production answer but needs a paid service and account. The env-var approach gives a stable identity across redeploys (so signatures keep verifying) at zero cost, while the runtime-generated key keeps local dev frictionless. The interface is HSM-ready — only the loader changes.
SchemaEnsureCreated on Postgres, migrations on SQLite
Decision
Branch at startup: Migrate() for SQLite, EnsureCreated() for Postgres.
Alternatives
Generate and maintain a parallel set of Postgres-specific migrations.
Why
EF migrations are provider-specific; the existing migrations were generated for SQLite and wouldn’t apply to Postgres. EnsureCreated builds the (simple) schema from the model on any provider — perfect for a demo. The acknowledged cost is no schema evolution on Postgres (logged as tech debt).
i18nA tiny custom service, not a library
Decision
A ~120-line I18n service with two dictionaries and a direction toggle.
Alternatives
ngx-translate (runtime) or @angular/localize (build-time).
Why
For two languages and a finite set of strings, a dependency-free service is lighter, instantly switchable at runtime, and trivial to reason about. It also directly drives dir="rtl", which is the part that actually matters for Arabic. @angular/localize would force one bundle per locale; ngx-translate adds a dependency for little gain at this size.
PDFQuestPDF for certificates
Decision
QuestPDF under its free Community licence; install libfontconfig1 in the Linux image.
Alternatives
PdfSharp (pure-managed, no native deps) or iText7 (AGPL).
Why
QuestPDF has by far the cleanest fluent layout API and produces a far nicer document. Its SkiaSharp engine needs one native library on Linux — a one-line apt-get. iText7’s AGPL licence is unsuitable for a freely-shared portfolio; PdfSharp’s cross-platform font story is fiddlier. The licence terms (free for individuals/small orgs) fit a portfolio.
CIUnit tests run locally, not in CI
Decision
CI builds both halves and publishes the image; Karma unit tests run locally (5 passing).
Alternatives
Run headless-Chrome Karma in CI with a custom no-sandbox launcher.
Why
GitHub runners can’t sandbox Chrome, and a custom karma.conf broke the test framework injection in the new Angular builder. A reliably green build gate was judged more valuable for a portfolio than a flaky red one. Tests still exist and run; re-enabling them in CI is a documented next step.
AuthSingle short-lived access token, no refresh
Decision
A 120-minute JWT stored in localStorage; re-login when it expires.
Alternatives
Refresh tokens + rotation; HttpOnly cookies.
Why
Refresh-token infrastructure (storage, rotation, revocation) is real work that doesn’t change the demo’s story. localStorage is simplest for an SPA; the XSS risk is mitigated by Angular’s default escaping. Both are flagged as the first hardening step for real use.
HostingSingle-origin (API serves the SPA)
Decision
In production the API serves Angular from wwwroot with a fallback to index.html.
Alternatives
A separate static host/CDN for the SPA + the API on its own origin (with CORS).
Why
One origin means no CORS, one URL, one deploy, and one free Render service. A CDN is the right move at scale (and is a listed next step), but adds moving parts the project doesn’t yet need.

5Next Steps & Recommendations

A prioritised roadmap for an engineer picking this up tomorrow — ordered by leverage.

Immediate (make it solid)

  1. Turn on production persistence. Sync the Render Blueprint (creates the free Postgres and wires DATABASE_URL) and set Signing__PfxBase64 + Signing__PfxPassword from deploy/New-SigningKey.ps1. Without this, cold starts wipe data and break old signatures.
  2. Add a backend test project. xUnit unit tests for RsaSigner (valid / tampered / wrong-owner) and integration tests with WebApplicationFactory + Testcontainers (real Postgres) for the sign/verify/delete endpoints. Re-enable Angular Karma in CI with a no-sandbox launcher.
  3. Harden auth. Refresh tokens with rotation + revocation, account lockout after repeated failures, and password reset.
  4. Validate uploads. Allow-list file types, verify the content type, and cap per-user storage.

Short term (make it production-grade)

  1. Real Postgres migrations. Replace EnsureCreated() with a provider-specific migration set so the schema can evolve safely.
  2. Structured logging & observability. Wire Serilog (JSON + correlation IDs), health checks, and OpenTelemetry traces/metrics.
  3. Move secrets & key to a vault / HSM. Source the signing key and JWT secret from a managed secret store; rotate on a schedule, keeping old public certs so historical signatures still verify (each signature already records its CertThumbprint).
  4. Object storage for uploads (S3 / Azure Blob) behind the existing IFileStore interface — a drop-in implementation.

Longer term (make it a trust product)

  1. PAdES embedded PDF signatures (via iText7 or a PAdES library) so signed PDFs verify natively in Adobe Reader.
  2. RFC 3161 trusted timestamping from an external authority.
  3. Tamper-evident audit log — hash-chain each row, or write to WORM storage; expose exportable audit reports.
  4. Admin dashboard using the already-modelled Admin role (user management, key rotation, audit review).
  5. Scale-out — front the API with a CDN for the SPA, add per-endpoint rate limits, and move to an always-on host to remove free-tier cold starts. Identify natural microservice seams (signing, verification, audit) only if traffic justifies it.
Where to start reading the code Program.cs (the composition root) → Services/ISigner.cs & SigningCertificate.cs (the crypto) → Controllers/DocumentsController.cs & VerifyController.cs (the use-cases). On the frontend, core/document.service.ts and features/verify/verify.ts show the whole client flow.

6Beginner’s Guide: Use, Run & Change It

No coding experience needed. This section explains, in plain language, how to get the project running and how to make small, safe changes.

What this app does (in one breath)

You log in, upload a file, and click Sign. The website stamps your file so it can’t be faked or secretly changed. Then anyone — even with no account — can check the file and see “Signed by you, authentic.” If someone edits the file, the check fails. That’s the whole idea.

The easiest way to see it: just open the website

It’s already online: signvault-q8mi.onrender.com. Sign in with the demo account — email demo@signvault.local, password Demo1234! — upload anything, and try it. (On the free plan the very first visit after a quiet period can take ~30–60 seconds to “wake up.” That’s normal.)

Run it on your own computer — from scratch

Step A — install the free tools (once)

1
.NET 10 SDK — runs the “engine” (backend). Get it from dotnet.microsoft.com/download.
2
Node.js (version 20 or newer) — runs the website (frontend). Get it from nodejs.org.
3
A code editor — Visual Studio Code is free and friendly: code.visualstudio.com.
4
One small extra: open a terminal and run npm install -g @angular/cli (installs the Angular tool). You don’t need to install any database — the app makes its own.

Step B — run the two halves (two terminals)

“Terminal” is the black text window where you type commands. In VS Code, open one with the Terminal menu → New Terminal. You need two of them.

Terminal 1 — the backend (the engine)
cd backend/SignVault.Api
dotnet run

Leave it running. It prints a line like Now listening on http://localhost:5080.

Terminal 2 — the frontend (the website)
cd frontend
npm install     # only needed the first time; downloads the building blocks
npm start

When it says it’s ready, open your browser to http://localhost:4200 and log in with the demo account above. That’s it — the app is running on your machine.

Even simpler, if you have Docker One command runs the whole thing with nothing else installed: docker run -p 8080:8080 ghcr.io/anas-lees/digital-signature-platform:latest, then open http://localhost:8080.

How to change things (safe, beginner tweaks)

The golden rule: change one small thing, save, and look at the browser — it updates by itself while the app is running. If something breaks, undo your change (Ctrl+Z) and save again.

A concrete first edit (try this)

  1. With the app running, open frontend/src/app/core/i18n.service.ts.
  2. Find 'verify.title': 'Check a document', and change it to 'verify.title': 'Is my document real?',.
  3. Save (Ctrl+S). Look at the Verify page in your browser — the heading changed instantly. Congratulations, you just edited the app.
Two things to avoid Don’t edit files inside node_modules, bin, obj, or publish — those are auto-generated. And never put real passwords or secret keys into files you might share; the app reads those from environment variables in production for exactly that reason.

How to check it still works after a change

  • The quick way: just use the app in the browser — upload, sign, and verify a file.
  • The thorough way (frontend): in the frontend folder run npm test — it runs the automated checks and prints “SUCCESS”.
  • If the backend won’t start: read the red text in Terminal 1 — it usually names the file and line to look at.
